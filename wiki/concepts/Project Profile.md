# Project Profile

A claude-mem design concept: a hand-curated, skill-assisted artifact that captures **calibration knowledge** about a brownfield codebase so that new code generated by an agent fits the host project rather than reads as generic. Status: design phase, informed by 2026-05-09 research pass.

## Origin

Came out of a discussion about Phase 2 (reconnaissance) and Phase 4 (implementation) of the agent's coding workflow. The user observation: brownfield projects with multi-year history have implicit conventions that an agent can't reliably extract from one quick scan, leading to plausible-looking-but-off code. The proposal: pre-compute a calibration artifact once, query it on every task.

## Research findings that reshape the design

The research pass on 2026-05-09 surfaced findings that **change** the obvious naive design:

1. **Auto-generation is net-negative.** ETH Zurich's AGENTbench (138 instances, 4 agents) found LLM-generated context files reduce task success by 2-3%. Human-written files give only +4%. Both options cost +20-23% inference and +3-4 extra steps. (Source: [[evaluating-agents-md-eth]])
2. **Repository overviews don't help.** The most commonly included section in AGENTS.md provides no measurable benefit.
3. **Specific actionable directives outperform descriptive prose.** When AGENTS.md mentions a tool or command, it gets used 1.6× more.
4. **Tribal knowledge is the only consistent winner.** Things that aren't in the code or README — that's what context files contribute.
5. **AGENTS.md is the de facto standard.** 60K+ projects, 20+ tools, Linux Foundation governance. Ignoring it loses interop.
6. **Hybrid is the consensus.** Anthropic, Aider, and the RACG survey all converge on: small upfront artifact + on-demand retrieval primitives.

## Revised design

### Output format: AGENTS.md-compatible, not novel

The skill should produce / update an AGENTS.md (or layered AGENTS.md files) at the repo root. Not a claude-mem-specific format. Reasons:

- Cross-tool interoperability is already established at scale.
- Sticking to spec means the same output helps Codex, Cursor, Copilot users on the same repo.
- Adding tool-specific extensions only when necessary (CLAUDE.md @path imports if the user uses Claude Code).

### What the skill should and should not generate

Based on AGENTbench findings:

| Section | Auto-generate? | Why |
|---|---|---|
| Repository overview | **NO** | Doesn't help, duplicates README |
| Build/test commands | **YES** if extractable from CI/Makefile | Concrete and actionable |
| Style enforcement (linter configs) | **YES** | Mechanical, accurate |
| Layer/architecture description | **NO** auto, **YES** human-prompted | Often wrong if guessed; valuable if curated |
| "Use X not Y" tribal knowledge | **NEVER** auto | Has to come from the human |
| "Do not touch" zones | **NEVER** auto | Same |
| Recent decisions / why | **NEVER** auto | Same |

### Skill workflow

```
1. Detect existing AGENTS.md / CLAUDE.md / .cursorrules
2. Run mechanical scans:
   - Linter / formatter configs
   - Test command from package.json / Makefile / pyproject.toml
   - Build command similarly
   - Tree-sitter pass for top-PageRank files (Aider-style repo-map, 1K token budget)
3. Show user a draft with mechanical sections filled in
4. Interview user for tribal sections:
   - "Anything I should never touch?"
   - "Tools/libraries: any 'use X not Y' rules?"
   - "What's being deprecated?"
   - "What's the current style for new code (vs legacy)?"
5. User edits / approves
6. Write AGENTS.md (root) and any per-subdirectory files
7. Cross-link with claude-mem wiki for narrative depth
```

### Refresh strategy

- Mechanical sections refresh automatically on `/project-profile --refresh`.
- Tribal sections require human re-interview.
- Recommend: refresh mechanical on commit-via-hook (cheap), tribal on a manual cadence (monthly?).

### Size / token budget

- AGENTS.md target: under ~2K tokens. Beyond that, AGENTbench data shows costs ramping without benefits.
- Repo-map artifact (separate file, optional): 1K tokens (Aider's default).
- Hot cache (already exists): ~500 words.
- Total upfront: <4K tokens. Below the 200K threshold where Anthropic recommends RAG; well within prompt-cache friendly range.

## Open design questions

- Do we ship layered AGENTS.md (root + subdirectory) or single? Spec allows both. Empirical paper tested only single.
- Does the skill produce a repo-map file alongside AGENTS.md, or only on demand? Aider produces it dynamically per-task.
- How do we capture and surface "current generation" of code in a multi-generation codebase? This is the brownfield-specific challenge that none of the literature solves. May need explicit user annotation (e.g., glob patterns marking "current style" zones).
- Verification cycle: when the agent says "this codebase uses X pattern," what gives the user a chance to correct it before the profile saves?

## Implementation prerequisites

Most exist already in claude-mem:

- Sub-agent dispatch pattern (graphify-extract-subagent, wiki-ingest-subagent)
- Tree-sitter / AST extraction (graphify)
- Wiki layer for narrative content
- Hot cache for session-priority context

Missing:

- AGENTS.md writer that respects format spec
- The structured user interview for tribal sections
- Refresh / drift detection
